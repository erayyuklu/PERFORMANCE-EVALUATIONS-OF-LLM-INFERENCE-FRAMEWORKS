# Planner-Executor Architecture for Cost Optimization

This document outlines the design and cost-optimization strategies for implementing a Planner-Executor (also known as "Big Brain, Small Hands") LLM architecture. This specific setup leverages an expensive, high-capability external API for complex reasoning (The Planner) and a self-hosted, highly-efficient vLLM deployment on GKE for task execution (The Executor).

## 1. Core Architecture: The Asymmetric Model Strategy

The fundamental principle of cost optimization in this architecture is routing tasks to the most cost-effective model capable of handling them.

*   **The Planner (External API - e.g., GPT-4o, Claude 3.5 Sonnet):** 
    *   **Role:** Acts as the "manager." Responsible for understanding complex user intents, breaking down massive goals into sequential or parallel sub-tasks, handling ambiguous reasoning, and final synthesis (if complex).
    *   **Cost Strategy:** Called infrequently. Used strictly for high-level cognitive tasks. Never used for rote data extraction, formatting, or simple summarization.
*   **The Executor (Self-hosted vLLM on GKE - e.g., Llama-3-8B, Qwen-2.5-7B):**
    *   **Role:** Acts as the "worker." Receives highly specific, well-defined prompts generated by the Planner. Executes tasks such as querying a database, parsing documents, formatting JSON, or generating standard text.
    *   **Cost Strategy:** Handles the overwhelming majority (e.g., 80-90%) of inference volume. Since it is self-hosted on GKE using vLLM, the marginal cost per token is close to zero (dependent only on underlying compute cluster costs), enabling massive scale at a fraction of the cost of external APIs.

## 2. Techniques for Cost Optimization

### A. Context Compression and Pruning
External APIs charge per token. If a multi-step workflow requires the Planner to be called multiple times, do not send the entire raw execution history.
*   **Action:** Have the cheaper Executor model summarize the results of its actions into concise "state updates" before feeding them back to the Planner.
*   **Benefit:** Drastically reduces the input token count for the expensive external API.

### B. Caching at the Planner Level (Semantic Caching)
Many user queries follow similar patterns. 
*   **Action:** Implement semantic caching (e.g., using Redis or a vector database) in front of the Planner. If a user asks a query conceptually similar to a previous one, return the previously generated execution plan.
*   **Benefit:** Bypasses the expensive external API entirely for recurring workflows, sending tasks directly to the self-hosted vLLM Executors.

### C. Parallel Execution of Sub-tasks
When the Planner decomposes a goal, some sub-tasks may be independent (e.g., "Summarize Doc A" and "Summarize Doc B").
*   **Action:** The Planner should explicitly flag tasks that can run concurrently. These are fanned out to the vLLM GKE cluster simultaneously.
*   **Benefit:** Takes full advantage of vLLM's continuous batching capabilities. While this optimizes latency more than direct cost, maximizing vLLM cluster utilization ensures you get the highest ROI on your GKE compute spend.

### D. Enforced Structured Outputs (JSON)
Relying on LLMs to extract information from messy conversational text is expensive and error-prone.
*   **Action:** The Planner must output its plan in a strict JSON schema. The vLLM Executors should also be forced to output JSON (e.g., using `guided_decoding` or `outlines` integrated with vLLM).
*   **Benefit:** Allows deterministic, traditional code (Python scripts) to handle the routing and parsing between the Planner and Executor, eliminating the need for an LLM to act as a "parser" or "router."

### E. Standardized "Meta-Tools"
Over time, you will notice the Planner generating the same sequence of sub-tasks repeatedly.
*   **Action:** Hardcode these recurring sequences into standard programmatic functions ("Meta-tools"). 
*   **Benefit:** When the Planner recognizes a common scenario, it invokes the Meta-tool (a single API token output) rather than generating a verbose, multi-step plan, saving significant external API token costs.

## 3. Deployment Considerations on GKE

To maximize the cost-efficiency of the Executor layer:
*   **Autoscaling based on Queue Length:** Configure GKE Horizontal Pod Autoscaler (HPA) to scale vLLM pods based on custom metrics (like the number of pending requests in the vLLM queue) rather than just CPU/GPU utilization.
*   **Right-sizing the Executor Model:** Continuously evaluate if a smaller, quantized model (e.g., AWQ or GPTQ 4-bit quantization) can handle the execution tasks. Smaller models require fewer GPUs and achieve significantly higher throughput on vLLM.
